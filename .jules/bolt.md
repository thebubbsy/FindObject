## 2024-05-24 - PowerShell Pipeline Pattern Optimization
**Learning:** PowerShell pipeline processing has significant per-object overhead. Specifically, generating dynamic strings like `"*$keyword*"` and executing `foreach` loops in the `process` block scales poorly. Type-casting by `.ToString()` on arbitrary objects can be slow and risky compared to `$objectName -isnot [string]`. State tracking variables (e.g., `$match`) add unnecessary instructions compared to using explicit early `return` statements. When converting a generic list `[System.Collections.Generic.List[string]]` to an array in PowerShell, calling `.ToArray()` directly can be unsafe or break if the underlying variable resolves to a scalar or standard array type; instead, use the array subexpression operator `@($list)` to safely ensure an array type.
**Action:** When filtering objects via pipeline, pre-calculate string manipulations (like wildcards) and cache lengths for arrays in the `begin` block. Safely construct arrays using `@($list)` rather than `.ToArray()`. In the `process` block, use `for` loops indexed over those lengths, use the `-isnot [string]` operator for type-checks instead of `.ToString()`, and leverage early returns over tracking states.
